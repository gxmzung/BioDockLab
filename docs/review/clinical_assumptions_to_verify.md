# BioDockLab 검증 필요한 의료/병원 업무 가정 목록

## 1. 현재 우리가 세운 가정

BioDockLab은 다음 가정을 기반으로 만들어졌습니다.

1. 환자, 간호사, 의사, 약사, 원무, 연구자, 보안관리자는 서로 다른 정보를 봐야 한다.
2. 환자는 전문적인 검사 결과보다 쉬운 설명이 필요하다.
3. 간호사는 바이탈, 인수인계, 투약 전 확인 정보가 필요하다.
4. 의사는 환자 요약과 결과 해석 리포트가 필요하다.
5. 약사는 처방량, 병용약, 약물상호작용, 유전자 적합성 정보를 봐야 한다.
6. 원무는 예약, 동의서, 서류 상태만 보면 된다.
7. 연구자는 실명 환자정보보다 비식별 연구 데이터가 필요하다.
8. 보안관리자는 의료 내용보다 접근 로그와 위험 이벤트를 봐야 한다.
9. 플랫폼 관리자는 역할별 권한 정책을 관리해야 한다.
10. 모든 접근 시도는 감사 로그로 남아야 한다.

## 2. 검증이 필요한 병원 시스템 용어

아래 용어는 정확한 이해가 필요합니다.

| 용어 | 현재 이해 | 검증 필요 사항 |
|---|---|---|
| EMR | 전자의무기록 | 간호기록, 처방, 검사결과가 어떤 식으로 구분되는지 |
| OCS | 처방전달시스템 | 의사 처방과 약사 검토 흐름 |
| HIS | 병원정보시스템 | 원무, 접수, 수납, 예약과의 관계 |
| 간호기록 | 간호사가 기록하는 환자 상태·처치 정보 | Nurse Workspace와 어떻게 구분해야 하는지 |
| PACS | 영상저장전송시스템 | 현재 프로젝트 범위에 포함하지 않아도 되는지 |
| LIS | 검사정보시스템 | 검사결과와 환자 설명 리포트의 관계 |
| CDSS | 임상의사결정지원시스템 | BioDockLab이 CDSS처럼 보이면 위험한지 |
| RBAC | 역할 기반 접근제어 | 병원 시스템에서 실제 적용 방식 |
| Audit Log | 접근 감사 로그 | 실제 의료기관에서 어느 수준으로 남기는지 |

## 3. 위험한 가정

다음 가정은 검토 없이 주장하면 위험합니다.

1. EMR보다 BioDockLab이 더 좋다.
2. AI가 환자에게 치료 방향을 설명해도 된다.
3. 간호사가 약물 유전자 적합성을 직접 판단한다.
4. 연구자가 환자 데이터를 쉽게 활용할 수 있다.
5. 환자에게 치료 후보 점수를 보여줘도 문제없다.
6. 보안 문제는 로그만 남기면 해결된다.
7. 응급실/구급차 문제도 BioDockLab이 바로 해결할 수 있다.
8. 병원에서 이런 시스템을 바로 도입할 수 있다.

## 4. 안전한 표현

아래 표현은 상대적으로 안전합니다.

- 설명 보조
- 상담 준비
- 연구 워크플로우
- 비식별 데이터
- 역할 기반 접근제어
- 민감정보 마스킹
- 접근 감사 로그
- 현장 검토가 필요한 초기 프로토타입
- EMR 대체가 아닌 보조 레이어
- 의료진 판단을 대체하지 않음

## 5. 교수님께 확인할 핵심 가정

1. 간호사 화면을 따로 두는 것이 타당한가?
2. 환자에게 보여주는 정보의 깊이는 어느 정도가 적절한가?
3. 약사 화면에서 유전자 적합성 정보를 다루는 것이 적절한가?
4. 원무 화면에서 동의서 상태를 보는 것이 현실적인가?
5. 연구자 화면은 병원 데이터와 분리하는 것이 맞는가?
6. 보안센터의 내부자 과다열람 탐지 개념은 현실성이 있는가?
7. EMR/OCS/HIS와 구분하기 위해 반드시 넣어야 할 문구는 무엇인가?
